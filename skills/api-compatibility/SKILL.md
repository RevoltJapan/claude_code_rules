# API Compatibility Skill

API互換性を検証するスキルです。
破壊的変更の検出、バージョン管理、後方互換性の確認を行い、APIの安定性と互換性を保証します。

## 使用タイミング

- API変更時（エンドポイント、リクエスト/レスポンス形式等）
- リリース前
- バージョンアップ時
- クライアントへの影響確認時

## チェック項目

### 1. 破壊的変更の検出

#### エンドポイント

- [ ] エンドポイントのURLが変更されていないか
- [ ] HTTPメソッドが変更されていないか
- [ ] エンドポイントが削除されていないか

#### リクエスト

- [ ] 必須パラメータが追加されていないか
- [ ] パラメータの型が変更されていないか
- [ ] パラメータ名が変更されていないか
- [ ] パラメータのバリデーションルールが厳格化されていないか

#### レスポンス

- [ ] レスポンス形式が変更されていないか
- [ ] 必須フィールドが削除されていないか
- [ ] フィールドの型が変更されていないか
- [ ] フィールド名が変更されていないか
- [ ] ステータスコードが変更されていないか

#### 認証・認可

- [ ] 認証方式が変更されていないか
- [ ] 認可要件が厳格化されていないか
- [ ] トークンの形式が変更されていないか

### 2. 後方互換性

- [ ] 既存のクライアントが引き続き動作するか
- [ ] 新規フィールドはオプショナルか（既存クライアントへの影響を最小化）
- [ ] 非推奨（deprecated）フィールドは適切な期間維持されるか
- [ ] 移行パスが提供されているか（破壊的変更が必要な場合）

### 3. バージョン管理

- [ ] APIバージョンが適切に管理されているか（URLパス、ヘッダー等）
- [ ] バージョン番号がSemVerに従っているか
- [ ] 破壊的変更がある場合はメジャーバージョンが上がっているか
- [ ] バージョンごとのサポート期間が明確か

### 4. ドキュメント

- [ ] APIドキュメントが更新されているか
- [ ] 破壊的変更が明記されているか
- [ ] 移行ガイドが提供されているか（破壊的変更がある場合）
- [ ] 変更履歴（CHANGELOG）が更新されているか

### 5. テスト

- [ ] 既存の統合テストが通過しているか
- [ ] 後方互換性のテストが実装されているか
- [ ] 破壊的変更のテストが実装されているか

## 破壊的変更の例

### エンドポイント

```diff
- GET /api/v1/users
+ GET /api/v1/members  # エンドポイント名の変更
```

### リクエスト

```diff
POST /api/v1/users
{
-  "name": "string"
+  "fullName": "string"  # フィールド名の変更
+  "email": "string"     # 必須フィールドの追加
}
```

### レスポンス

```diff
{
-  "id": "string"
+  "userId": "string"    # フィールド名の変更
-  "status": "string"
+  "status": "number"    # 型の変更
}
```

## 互換性戦略

### 1. 後方互換性を維持

- 新規フィールドはオプショナルにする
- 既存フィールドは削除しない
- 非推奨マーカーを使用して段階的に移行

### 2. バージョニング

- URLパス: `/api/v1/`, `/api/v2/`
- ヘッダー: `API-Version: 1.0`
- クエリパラメータ: `?version=1.0`

### 3. 非推奨プロセス

1. 非推奨マーカーを追加（deprecated）
2. ドキュメントに移行ガイドを記載
3. 適切な期間（例: 6ヶ月）維持
4. 次のメジャーバージョンで削除

## 出力形式

```

- [変更内容]: [影響範囲と推奨対応]

#### 必須対応
- [変更内容]: [影響範囲]

#### ドキュメント更新
- [変更内容]: [更新すべきドキュメント]

### 📝 推奨事項
- [改善提案]
```

## バージョン管理戦略

### SemVerに基づくAPIバージョニング

- **MAJOR**: 破壊的変更（後方互換性なし）
- **MINOR**: 新機能追加（後方互換性あり）
- **PATCH**: バグ修正（後方互換性あり）

### 例

```
v1.0.0 → v1.1.0: 新機能追加（互換性あり）
v1.1.0 → v2.0.0: 破壊的変更（互換性なし）
v2.0.0 → v2.0.1: バグ修正（互換性あり）
```

## テスト戦略

### 1. 契約テスト（Contract Testing）

- APIの契約（スキーマ）を定義
- 変更時に契約違反を検出
- ツール: Pact、OpenAPI等

### 2. 回帰テスト

- 既存のクライアントシナリオをテスト
- 後方互換性を確認

### 3. 統合テスト

- エンドツーエンドの動作確認
- 実際のクライアントとの互換性確認

## 注意事項

- 破壊的変更は可能な限り避ける
- やむを得ない場合は適切な移行期間を設ける
- クライアントへの影響を最小限に抑える
- 変更は必ずドキュメント化する

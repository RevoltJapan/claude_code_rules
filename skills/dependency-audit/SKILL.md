# Dependency Audit Skill

依存関係の包括的な監査を実行するスキルです。
セキュリティ脆弱性、ライセンス、バージョン管理、不要な依存関係を検証し、プロジェクトの依存関係を健全に保ちます。

## 使用タイミング

- 新規依存パッケージ追加時
- 定期的な監査時（週次/月次推奨）
- リリース前
- セキュリティアラート受信時

## チェック項目

### 1. セキュリティ脆弱性

- [ ] 既知のセキュリティ脆弱性がないか（CVE、npm audit、go mod、等）
- [ ] 脆弱性の重大度レベルを確認（Critical/High/Medium/Low）
- [ ] 脆弱性の修正方法を確認（パッチ、バージョンアップ等）
- [ ] 脆弱性の影響範囲を確認（直接依存/間接依存）

#### 脆弱性レベル

- **Critical**: 即座に修正が必要。本番環境への影響が大きい
- **High**: 優先的に修正が必要。可能な限り早期に対応
- **Medium**: 計画的な修正が必要。次回リリースまでに対応
- **Low**: 改善提案。可能な範囲で対応

### 2. ライセンス

- [ ] すべての依存パッケージのライセンスが確認されているか
- [ ] ライセンスがプロジェクトのライセンスポリシーに準拠しているか
- [ ] コピーレフトライセンス（GPL等）の使用が適切に管理されているか
- [ ] ライセンスファイルが適切に含まれているか（必要に応じて）

#### ライセンスカテゴリ

- **Permissive**: MIT、Apache-2.0、BSD等（使用推奨）
- **Weak Copyleft**: LGPL、MPL等（使用可能、注意が必要）
- **Strong Copyleft**: GPL等（使用前に法的確認が必要）
- **Proprietary**: 商用ライセンス（使用前に承認が必要）

### 3. バージョン管理

- [ ] 依存パッケージのバージョンが固定されているか（package-lock.json、go.sum等）
- [ ] セマンティックバージョニングに従っているか
- [ ] 最新の安定版を使用しているか（可能な範囲で）
- [ ] 非推奨（deprecated）パッケージを使用していないか

### 4. 依存関係の健全性

- [ ] 不要な依存関係が含まれていないか
- [ ] 重複する依存関係がないか
- [ ] 依存関係のツリーが適切な深さか（過度に深くないか）
- [ ] 依存関係のサイズが適切か（バンドルサイズへの影響）

### 5. メンテナンス状況

- [ ] パッケージが積極的にメンテナンスされているか
- [ ] 最終更新日が最近か（1年以上更新がない場合は注意）
- [ ] 既知の問題やバグがないか
- [ ] 代替パッケージの検討が必要か

### 6. プロジェクト固有の要件

- [ ] 依存パッケージがプロジェクトの技術スタックと互換性があるか
- [ ] 依存パッケージがプロジェクトのサポート対象環境と互換性があるか
- [ ] 依存パッケージの追加がプロジェクトの方針に沿っているか

## 監査ツール

### 言語別の推奨ツール

#### Node.js / npm
- `npm audit` - セキュリティ脆弱性チェック
- `npm outdated` - 古いパッケージの検出
- `license-checker` - ライセンス確認

#### Go
- `go list -m -u all` - 更新可能なモジュール確認
- `golang.org/x/vuln/cmd/govulncheck` - 脆弱性チェック
- `go-licenses` - ライセンス確認

#### Python
- `pip-audit` - セキュリティ脆弱性チェック
- `pip list --outdated` - 古いパッケージの検出
- `pip-licenses` - ライセンス確認

#### Ruby
- `bundle audit` - セキュリティ脆弱性チェック
- `bundle outdated` - 古いパッケージの検出

## 対応フロー

### 脆弱性発見時

1. **Critical/High**: 即座に修正計画を立てる
   - パッチが利用可能か確認
   - バージョンアップが可能か確認
   - 代替パッケージの検討（必要に応じて）

2. **Medium**: 次回リリースまでに対応計画を立てる

3. **Low**: 改善計画に追加（優先度低）

### ライセンス問題発見時

1. ライセンスポリシーに照らして評価
2. 法的確認が必要な場合は法務部門に相談
3. 問題がある場合は代替パッケージを検討

## 自動化の推奨

- CI/CDパイプラインに依存関係監査を組み込む
- 週次/月次で自動監査を実行
- セキュリティアラートを自動通知

## 注意事項

- 依存関係の更新は慎重に行う（破壊的変更の可能性）
- 本番環境への反映前に十分なテストを実施
- ライセンス問題は法的リスクを伴うため、専門家に相談

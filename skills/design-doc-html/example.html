ログイン

パスワード照合は 必ず bcrypt.compare を使い、自前比較・タイミング攻撃に脆弱な実装は禁止する。

背景 / 目的 / 利用者

登録済みユーザーがアプリケーションにログインするためのエンドポイント。 現在は仮認証（環境変数のトークン）で運用しているため、本実装で置き換える。

入力

項目型必須制約
emailstring必須RFC5322、最大 254 文字
passwordstring必須8〜128 文字、平文で受領

処理

  1. email でユーザーを検索する。見つからない場合は 401 を返す（存在判定を露出しない）。
  2. bcrypt.compare(password, user.password_hash) で照合する。
  3. 照合成功時、{ sub: user.id, exp: now + 24h } の JWT を発行する（D-001）。
  4. 発行した JWT を access_token としてレスポンスする。
  5. 失敗時は 401 Unauthorized を返し、レート制限カウンタを進める（Q-001 の方針確定後）。

出力

成功200 { "access_token": "<jwt>", "token_type": "Bearer", "expires_in": 86400 }
失敗401 { "error": "invalid_credentials" }

例外 / エッジケース

ケース挙動
ユーザーが存在しない401。存在判定を露出させない
パスワード照合失敗401。同上
退職フラグ is_active=false のユーザー401。理由は返さない
password_hash が NULL（外部 IdP 連携ユーザー）401。本フローでは認証不可
email の大文字小文字差小文字化して比較する

スコープ外（実装しないこと）

  • パスワード再発行フロー — 別タスクで扱う
  • ソーシャルログイン（Google / GitHub） — Phase 2 で扱う
  • 二要素認証 (TOTP / SMS) — 本リリースでは対象外
  • リフレッシュトークン — D-003 により本フェーズでは導入しない
  • アカウント凍結処理 — 別機能で扱う

ログアウト

背景 / 目的 / 利用者

JWT は本来クライアント側破棄で完結するが、サーバー側ブラックリストで強制無効化できるようにする。

入力

項目型必須制約
Authorization ヘッダstring必須Bearer <jwt> 形式

処理

  1. JWT の jti（または signature ハッシュ）をブラックリストテーブルに記録する。有効期限は元 JWT の exp まで。
  2. 204 No Content を返す。

出力

204 No Content

例外 / エッジケース

ケース挙動
すでにブラックリスト登録済み冪等に 204 を返す
JWT 期限切れ204。すでに無効なので何もしない

スコープ外

  • 全デバイス一括ログアウト — Phase 2 で扱う

トークン検証ミドルウェア

背景 / 目的 / 利用者

保護されたエンドポイントの直前に挟み、JWT を検証して req.user に展開する。

入力

項目制約
Authorization ヘッダBearer <jwt>

処理

  1. ヘッダ欠如時は 401。
  2. JWT 署名検証。失敗時は 401。
  3. 期限切れは 401 { "error": "token_expired" }。
  4. ブラックリストに存在する場合は 401 { "error": "token_revoked" }。
  5. 問題なければ req.user = { id: sub } を設定して next() を呼ぶ。

出力

成功時は req.user を埋めて後段ハンドラに委譲。失敗時は上記 401 いずれか。

例外 / エッジケース

ケース挙動
JWT は有効だが sub のユーザーが削除済み401 { "error": "user_not_found" }

スコープ外

  • 権限（ロール / スコープ）チェック — 別ミドルウェアで実装する

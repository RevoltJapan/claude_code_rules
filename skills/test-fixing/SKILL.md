# Test Fixing Skill

失敗したテストを分析し、修正方法を提案するスキルです。
テストの失敗原因を特定し、コードの修正、テストの修正、またはテストデータの更新を提案します。

## 使用タイミング

- テスト実行後に失敗が検出された時
- CI/CDパイプラインでテストが失敗した時
- リファクタリング後にテストが失敗した時
- フレーキーテスト（不安定なテスト）が検出された時

## 失敗原因の分析

### 1. エラータイプの分類

#### Assertion Error（アサーション失敗）

- [ ] 期待値と実際の値が一致していない
- [ ] アサーションの条件が正しいか
- [ ] テストデータが適切か

**例**:
```
Expected: "Hello World"
Actual: "Hello"
```

#### Runtime Error（実行時エラー）

- [ ] NullPointerException / TypeError
- [ ] インデックス範囲外エラー
- [ ] 型エラー
- [ ] 未定義の変数/関数

**例**:
```
TypeError: Cannot read property 'name' of undefined
```

#### Timeout Error（タイムアウト）

- [ ] テストが時間内に完了しない
- [ ] 非同期処理の待機が不適切
- [ ] 無限ループの可能性

#### Setup/Teardown Error（セットアップ/クリーンアップエラー）

- [ ] テストデータの準備に失敗
- [ ] リソースのクリーンアップに失敗
- [ ] モックの設定が不適切

### 2. 原因の特定

#### コードの変更による影響

- [ ] 関数/メソッドのシグネチャ変更
- [ ] 戻り値の型や形式の変更
- [ ] エラーハンドリングの変更
- [ ] 動作の変更（意図的/非意図的）

#### テストの問題

- [ ] テストケースが古い（コード変更に追従していない）
- [ ] アサーションが不適切
- [ ] テストデータが不適切
- [ ] モック/スタブが不適切

#### 環境の問題

- [ ] テスト環境の設定が不適切
- [ ] 外部依存（DB、API等）の問題
- [ ] タイムゾーンやロケールの問題

#### タイミングの問題

- [ ] 非同期処理の待機が不適切
- [ ] レースコンディション
- [ ] フレーキーテスト（不安定なテスト）

## 修正方法の提案

### 1. コードの修正が必要な場合

#### バグの修正

- [ ] エラーハンドリングの追加
- [ ] 境界値チェックの追加
- [ ] 型安全性の向上

#### 動作の変更

- [ ] 意図的な変更: テストを更新
- [ ] 非意図的な変更: コードを修正

### 2. テストの修正が必要な場合

#### テストケースの更新

- [ ] 期待値の更新
- [ ] アサーションの修正
- [ ] テストデータの更新

#### テスト構造の改善

- [ ] Arrange-Act-Assertパターンの適用
- [ ] テストの分離（独立性の確保）
- [ ] テスト名の明確化

### 3. モック/スタブの更新

- [ ] モックの戻り値を更新
- [ ] モックの呼び出し回数を確認
- [ ] スタブの動作を更新

### 4. テストデータの更新

- [ ] テストデータの準備方法を修正
- [ ] テストデータのクリーンアップを追加
- [ ] テストデータの整合性を確認

## 修正プロセス

### 1. 失敗の再現

- [ ] 失敗したテストを再実行
- [ ] 失敗が一貫して再現するか確認
- [ ] フレーキーテストかどうか判断

### 2. 原因の特定

- [ ] エラーメッセージを詳細に分析
- [ ] スタックトレースを確認
- [ ] 関連するコードを確認
- [ ] 最近の変更を確認

### 3. 修正の実装

- [ ] 最小限の変更で修正
- [ ] 修正後のテスト実行
- [ ] 他のテストへの影響を確認

### 4. 検証

- [ ] 修正したテストが成功するか確認
- [ ] 既存のテストが引き続き成功するか確認
- [ ] コードレビューを実施

## フレーキーテストの対処

### フレーキーテストの特徴

- 時々失敗する（再現性が低い）
- タイミングに依存する
- 環境に依存する
- 並列実行時に失敗する

### 対処方法

1. **再試行メカニズムの追加**
   - テストフレームワークのretry機能を使用
   - カスタムリトライロジックを実装

2. **テストの安定化**
   - タイミング依存を排除
   - 適切な待機処理を追加
   - テストの分離を改善

3. **環境の安定化**
   - テスト環境の設定を統一
   - 外部依存をモック化
   - テストデータの整合性を保証

## 注意事項

- テストの失敗は必ず修正する（無視しない）
- フレーキーテストは早期に修正する
- テストの修正は慎重に行う（テストの意図を理解してから）
- コードの修正とテストの修正を適切に使い分ける

# Test Runner Skill

テストの実行と結果分析を自動化するスキルです。
単体テスト、統合テスト、E2Eテストを実行し、結果を分析して、失敗したテストの原因を特定し、修正方法を提案します。

## 使用タイミング

- 実装完了後（`/impl` コマンドのPhase 3）
- コードレビュー時（`/review` コマンド実行時）
- リファクタリング後
- マージ前
- CI/CDパイプライン（自動実行）

## テスト実行の種類

### 1. 単体テスト（Unit Tests）

- [ ] すべての単体テストが実行されているか
- [ ] テストが適切に分離されているか（外部依存のモック化）
- [ ] テストが高速に実行されるか（推奨: 数秒以内）
- [ ] テストが独立しているか（実行順序に依存しない）

### 2. 統合テスト（Integration Tests）

- [ ] 統合テストが実行されているか
- [ ] データベースや外部サービスとの統合がテストされているか
- [ ] テストデータが適切に管理されているか（セットアップ/クリーンアップ）
- [ ] テスト環境が適切に設定されているか

### 3. E2Eテスト（End-to-End Tests）

- [ ] E2Eテストが実行されているか（該当する場合）
- [ ] 主要なユーザーフローのテストが含まれているか
- [ ] テスト環境が本番環境に近い状態か

### 4. パフォーマンステスト

- [ ] パフォーマンステストが実行されているか（該当する場合）
- [ ] レスポンスタイムが基準を満たしているか
- [ ] スループットが基準を満たしているか

## テスト実行コマンド

### 言語・フレームワーク別の推奨コマンド

#### Node.js / JavaScript / TypeScript
```bash
# Jest
npm test
npm test -- --coverage
npm test -- --watch

# Vitest
npm run test
npm run test:coverage

# Mocha
npm test
```

#### Go
```bash
# すべてのテスト
go test ./...

# カバレッジ付き
go test -cover ./...

# ベンチマーク
go test -bench=.

# 詳細出力
go test -v ./...
```

#### Python
```bash
# pytest
pytest
pytest --cov
pytest --cov-report=html

# unittest
python -m unittest discover
```

#### Ruby
```bash
# RSpec
bundle exec rspec
bundle exec rspec --format documentation

# Minitest
bundle exec rake test
```

## チェック項目

### 1. テスト実行結果

- [ ] すべてのテストが成功しているか
- [ ] テストの実行時間が適切か（タイムアウトがないか）
- [ ] テストの実行順序が適切か（並列実行可能か）

### 2. 失敗したテストの分析

- [ ] 失敗したテストの原因を特定
- [ ] エラーメッセージが明確か
- [ ] スタックトレースを分析
- [ ] 再現手順を確認

### 3. テストカバレッジ

- [ ] テストカバレッジが基準を満たしているか（プロジェクト基準に従う、推奨: 80%以上）
- [ ] 新規コードのテストカバレッジが基準を満たしているか
- [ ] カバレッジレポートが生成されているか

### 4. テストの品質

- [ ] テストが意味のあるアサーションを含んでいるか
- [ ] テストが適切に名前付けされているか
- [ ] テストが適切に構造化されているか（Arrange-Act-Assert）
- [ ] テストが適切にドキュメント化されているか（必要に応じて）

### 5. テスト環境

- [ ] テスト環境が適切に設定されているか
- [ ] テストデータが適切に管理されているか
- [ ] 外部依存が適切にモック化されているか
- [ ] テストが冪等性を保っているか（何度実行しても同じ結果）

## 失敗したテストの分析

### 1. エラータイプの分類

- **Assertion Error**: アサーション失敗（期待値と実際の値の不一致）
- **Runtime Error**: 実行時エラー（NullPointerException、TypeError等）
- **Timeout Error**: タイムアウト（テストが時間内に完了しない）
- **Setup/Teardown Error**: セットアップ/クリーンアップエラー

### 2. 原因の特定

- コードの変更による影響
- テストデータの問題
- 環境設定の問題
- 外部依存の問題
- タイミングの問題（非同期処理）

### 3. 修正方法の提案

- コードの修正が必要か
- テストの修正が必要か
- テストデータの更新が必要か
- モックの更新が必要か

## テスト実行戦略

### 1. 段階的実行

1. **単体テスト**: 最初に実行（高速、早期フィードバック）
2. **統合テスト**: 単体テスト成功後に実行
3. **E2Eテスト**: 統合テスト成功後に実行（時間がかかるため）

### 2. 並列実行

- 可能な限り並列実行して実行時間を短縮
- テスト間の依存関係がないことを確認

### 3. 選択的実行

- 変更されたファイルに関連するテストのみ実行（watch mode）
- 影響範囲を分析して関連テストを特定

## テストの再実行

### 自動再実行

- 失敗したテストのみ再実行（retry機能）
- フレーキーテストの検出と再実行

### 手動再実行

- 修正後の再実行
- 特定のテストケースのみ実行

## CI/CD統合

### 推奨設定

- すべてのPRでテストを自動実行
- テスト失敗時はマージをブロック
- テストカバレッジが基準未満の場合は警告

## 注意事項

- テストは常に最新のコードに対して実行する
- テスト環境は本番環境に近い状態を保つ
- フレーキーテスト（不安定なテスト）は早期に修正する
- テストの実行時間を監視し、最適化する
